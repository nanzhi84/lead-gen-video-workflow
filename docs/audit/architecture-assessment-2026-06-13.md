# cutagent-genesis 架构评估报告

> 13 个 Opus 子代理并行评审（8 子系统 × 三维 + 3 跨切面镜头 + 综合），只读取证。生成于 2026-06-13。

## 总评

| 维度 | 评级 |
|---|---|
| 性能 Performance | **C+** |
| 可扩展性 Scalability | **C-** |
| 可维护性 Maintainability | **B-** |

**执行摘要：** cutagent-genesis is a genuinely well-architected clean rewrite: a Pydantic-v2 single-source-of-truth contract layer that mechanically generates openapi.json + schema.d.ts (CI-enforced via git diff --exit-code), a near-perfectly inward-pointing dependency graph, durable Temporal-per-node execution with content-addressed tiered OSS storage, and a clean object-store/secret-store layer. The skeleton can scale. BUT the system as currently wired is NOT safe to run at its stated operational target (multi-worker Temporal + multi-replica API). The single most important issue, raised independently by 8 of 12 reviewers, is that one process-global, lock-free in-memory Repository (50+ plain dicts) is shared across an 8-thread activity pool AND wired into the gateway/registry/workflow even when CUTAGENT_STORAGE_BACKEND=sqlalchemy — concurrent runs interleave reads/writes on the same dicts (cross-run data bleed) and the dicts are never pruned (unbounded memory growth). The second thing that matters most is a cluster of multi-process correctness gaps that silently break at >1 worker/replica: a node-local ephemeral OSS tier (pipeline fails across workers), an outbox dispatcher with no SELECT...FOR UPDATE SKIP LOCKED (double-published events / lost live tail), no provider-call rate limiting despite a persisted concurrency_key, and per-process Prometheus registries. None of these are caught by the green suite because the SQL + Temporal paths — the actual production configuration — are gated behind env flags and skipped by default.

**评级依据：** Performance C+: the hot path is dominated by per-node full hydrate+sync (~16 hydrate + ~16 sync per run, each re-loading ALL provider_profiles/voices unfiltered and re-scanning ~10 in-memory dicts), full-object-into-RAM OSS I/O on minutes-long video, the hottest query (artifacts WHERE run_id) unindexed, and two full -count_frames decode passes per run used only as gates — all real but mechanical to fix. Scalability C- (resolving reviewer spread of D→C: the design CAN scale because Postgres+Temporal are the true system of record, but TODAY four concrete defects block safe multi-worker/multi-replica operation: shared lock-free Repository, node-local ephemeral tier, unsynchronized outbox dispatcher, unbounded provider fan-out; these are correctness blockers, not tuning knobs, hence below C). Maintainability B-: strong spine (enforced contract SSOT, clean layering, consistent error taxonomy, zero TODO debt) genuinely earns a B, pulled down by the ~259-branch dual-path tax, three 2000+ LOC god-files, a core<->production layering cycle, and the test gap on the production config.

## 架构优势（值得保留）

- Contract single-source-of-truth is enforced mechanically, not by convention: Pydantic v2 contracts generate openapi.json + schema.d.ts and CI fails on git diff --exit-code (scripts/ci_gate.sh), with frontend client.ts/r6.ts deriving every request/response type from the generated schema and tests/contracts/*.typecheck.ts giving compile-time API coverage — backend/frontend type drift is structurally impossible.
- The object-store + secret-store layer is genuinely production-grade in isolation: a tight ObjectStore ABC with faithful Local/S3 impls, content-addressed sha256 skip-if-exists (concurrency-safe by construction — identical bytes converge to the same key), tiered durable/ephemeral routing that refuses a shared-bucket misconfig, SigV4 presigned URLs, and a SecretStore with envelope-at-rest + path-traversal guard and no plaintext on any model.
- Durable execution is correctly modeled: minimal {run_id,node_id} payloads cross the Temporal boundary (heavy media stays in PG/OSS so history stays small and replay is cheap), per-node tiered timeouts, signal-based cooperative cancel checked at node boundaries, and resume that reuses a validated artifact prefix (compute_reuse_plan) — workflow id = run_id gives natural per-run dedupe.
- Near-perfect inward dependency direction: core/contracts is a clean leaf, packages never import from apps/, and a tight 4-method WorkflowRuntimeAdapter Protocol cleanly abstracts local-vs-Temporal so node logic is never forked between backends — the pipeline runs identically under both runtimes.
- Disciplined schema and hygiene: 50-table typed-Mapped schema with MetaData naming conventions, FK ondelete policies, CheckConstraints, a correctly-covering idx_outbox_pending matching the dispatcher query, DB-level UniqueConstraints providing real idempotency primitives (outbox dedupe, selection_ledger), plus zero TODO/FIXME debt and a recent de-lint of 464 unused imports.
- The ops/observability and ffmpeg layers are clean and well-factored: structured JSON logging with async-safe ContextVar context propagation, a coherent domain-aligned Prometheus metric vocabulary, and a single FfmpegRunner with typed errors, careful path escaping, and timeout→ErrorCode mapping.

## 关键风险

### 1. [CRITICAL · scalability/correctness] One process-global, lock-free in-memory Repository (50+ plain dicts) is shared across the worker's 8-thread activity pool AND wired into provider_gateway/prompt_registry/workflow even when the SQL backend is active. Concurrent run_node activities for different runs all hydrate into and mutate the SAME repository.artifacts/runs/node_runs/provider_invocations/provider_profiles dicts with zero synchronization, and nothing prunes them after a run completes.
- **证据**：apps/worker/main.py:35 single Repository() -> ProviderGateway (43) + TemporalActivityContext (59); activity_executor=ThreadPoolExecutor(max_workers=8) (73); apps/api/app.py:103-104,150-170 same shared instance under SQL backend; repository.py has plain dicts and no locks; hydrate writes GLOBAL-keyed provider_profiles/voices (sqlalchemy_repository.py:646-651) and per-run dicts (655-684) with no eviction.
- **影响**：Cross-run data bleed and intermittent invocation/usage-accounting corruption under any concurrency >1 run/worker, plus an unbounded memory leak that grows for the worker's whole lifetime and makes every per-node dict scan slower. This is the single load-bearing barrier to running more than one concurrent run safely. Raised independently by 8 of 12 reviewers (highest-confidence finding in the report).

### 2. [CRITICAL · scalability] The ephemeral OSS tier defaults to a NODE-LOCAL temp directory, so an intermediate (portrait track / lipsync / rendered) written by one worker is unreadable by the next node's activity on another worker. Temporal freely schedules consecutive activities on different workers, so multi-worker pipelines fail with artifact_missing unless ephemeral is explicitly pointed at shared MinIO/S3.
- **证据**：object_store_env.py:52-60 — _ephemeral_store_from_env defaults CUTAGENT_EPHEMERAL_OBJECTSTORE_BACKEND='local' rooted at tempfile.gettempdir()/'cutagent-ephemeral'; PortraitTrackBuild writes tier='ephemeral' (digital_human.py:2007-2012) and downstream nodes read back via _artifact_path -> local_object_path (assets.py:9-17).
- **影响**：The pipeline is silently single-worker-only in its default config. The moment a second worker is added, runs fail mid-pipeline. Must fail-fast at startup if ephemeral resolves local while runtime=temporal.

### 3. [CRITICAL · scalability] No backpressure or rate-limiting on provider calls. ProviderProfile.concurrency_key and timeout_sec are seeded and persisted but never enforced — there is no semaphore, token bucket, or in-flight cap anywhere in packages/ai. With 8 threads x M workers, concurrent runs fan out unbounded simultaneous TTS/ASR/VLM/LipSync/LLM calls straight at vendor quotas.
- **证据**：ProviderGateway.invoke dispatches synchronously with no concurrency control (provider_gateway.py:145-260); grep for Semaphore/rate_limit in packages/ai returns nothing; concurrency_key only copied through (gateway/sqlalchemy_repository.py:46,201). Confirmed directly.
- **影响**：quota_exceeded (429) becomes the de-facto rate limiter under load — exactly the failure mode the sandbox models. Compounded by all long-poll providers blocking a worker thread with time.sleep (runninghub.py:225, dashscope.py:274) for up to minutes, saturating the 8-slot pool on sleeping threads.

### 4. [HIGH · scalability] Realtime event delivery is single-instance. The fanout hub is in-process and each API replica runs its own SqlAlchemyOutboxDispatcher with NO SELECT...FOR UPDATE SKIP LOCKED. With N replicas, dispatchers race the same pending rows; a websocket subscriber connected to a different replica than the one that claimed/published an event never receives the live tail (historical replay still works).
- **证据**：InProcessFanoutHub holds subscribers in process memory (events.py:19-37); dispatch_once selects pending and flips status='published' with no row locking (events.py:138-168, confirmed); EventStreamTokenStore is also in-process so tokens issued by replica A fail on replica B.
- **影响**：Live run updates silently break at >1 API replica. Needs a shared bus (Redis pub/sub or PG LISTEN/NOTIFY) + FOR UPDATE SKIP LOCKED, or a singleton dispatcher sidecar.

### 5. [HIGH · performance] Per-node full hydrate+sync is the dominant hot-path cost, and the hottest query is unindexed. Every one of ~16 node activities re-loads the run from SQL — including unconditional full-table SELECTs of ALL provider_profiles and ALL voices — and re-syncs by re-scanning ~10 in-memory dicts plus ~10 session.flush() barriers. The artifacts WHERE run_id query (run 16x/run + every API read) cannot use any index because the only artifacts index leads with case_id.
- **证据**：hydrate unfiltered select(ProviderProfileRow)/select(VoiceProfileRow) (sqlalchemy_repository.py:646-651, confirmed); sync re-merges all run collections with repeated flushes (524-584); ArtifactRow.run_id is plain String with NO standalone index (database.py:162) and only idx_artifacts_case_run leads with case_id (792) — contrast NodeRunRow.run_id which IS indexed (793).
- **影响**：Per-node DB cost grows with the global catalog and the worker's accumulated state, not the run; the artifacts table reaches 6-figure rows and the central lookup degrades to seq scans. Adding Index(idx_artifacts_run) and scoping hydrate are high-leverage, low-risk.

### 6. [HIGH · performance] All media I/O fully buffers whole objects in memory. store_file does path.read_bytes() then put_bytes wraps it in io.BytesIO; get_bytes downloads the entire object into a BytesIO and returns .getvalue(); _path materializes the full object before caching. For minutes-long hundreds-of-MB videos x 8 concurrent activities this is multi-GB transient heap per worker, and it defeats the already-configured multipart TransferConfig.
- **证据**：object_store.py:191-212 (put_bytes/get_bytes), :243-249 (_path); packages/media/assets.py:28 path.read_bytes(). ~7 uploads/run. Confirmed directly.
- **影响**：Direct OOM / memory-pressure path under the stated workload. The fix (boto3 upload_file/download_file by path into the existing disk cache) is one method away since the cache already writes to disk.

### 7. [HIGH · correctness] Node re-execution is not idempotent and heartbeat death is undetectable. A Temporal-driven retry of run_node unconditionally appends a NEW NodeRun and re-runs side-effecting handlers with no 'already-succeeded -> skip' guard; NodeRun.attempt is never incremented so retry metrics under-report. Separately, activity.heartbeat is emitted but NO heartbeat_timeout is configured, so a worker dying mid-LipSync is undetectable until the 120-minute start_to_close ceiling.
- **证据**：digital_human.py:551 NodeRun(...) then :568 node_runs[run.id].append(node_run) with no existing-succeeded check; activity.heartbeat at temporal_adapter.py:163,166 but zero heartbeat_timeout anywhere (confirmed); LipSync timeout 120*60s.
- **影响**：Retries leave duplicate NodeRun rows, corrupt retry telemetry, and re-invoke PAID providers; a crashed worker blocks a run for up to 2 hours.

### 8. [HIGH · maintainability] The production configuration is not exercised by the default test suite. The SQL path and Temporal runtime — the actual operational config — are gated behind CUTAGENT_RUN_DB_TESTS=1 / CUTAGENT_RUN_TEMPORAL_TESTS=1 and skipped in the everyday ~237-268 green run, which exercises only the in-memory + sandbox path. The 73 row<->contract mappers and 52 ORM rows that must track 262 contracts run only in opt-in CI, and there is no parity test asserting the two backends agree.
- **证据**：tests/integration/test_sqlalchemy_production.py:7-8 pytest.skip allow_module_level; tests/temporal/test_temporal_runtime.py:17-20 skipif (both confirmed). rg SqlAlchemyProductionRepository in tests outside integration is empty.
- **影响**：SQL-only divergences and mapper omissions ship unnoticed; the green suite gives false confidence about the path that actually runs in production. This is why the critical concurrency/wiring bugs above survive.

### 9. [MEDIUM · maintainability] Concentrated change-risk: three 2000+ LOC god-files plus a ~259-branch dual-path tax and a core<->production layering cycle. digital_human.py (2439 LOC) fuses orchestration + all 16 node handlers + ffmpeg command construction + OSS I/O + GC; contracts/__init__.py (262 classes, 2087 LOC) is a merge-conflict hotspot harboring real defects (OutboxEvent.dedupe_key silently redefined nullable, ArtifactKind 43 names collapsing to 33 with 10 silent aliases, 4 artifact classes defined twice with conflicting fields). The dual-path SQL/in-memory design forces ~259 'is/None' branches across service modules.
- **证据**：wc -l confirms 2439/1702/2087; OutboxEvent.dedupe_key.is_required()==False and ArtifactKind 33-vs-43 confirmed at runtime; grep totals 259 is/None branches across apps/api/services; temporal_adapter.py:24,351 imports up into production (cycle broken only by Temporal sandbox + a deferred import).
- **影响**：High change-amplification (a persisted feature touches ~13 files/6 layers), latent correctness bugs hidden by file size (the weakened outbox idempotency key is invisible in a 2087-line file), and easy SQL/in-memory behavioral drift.

## 优先级行动清单

### 🔴 立刻
- **Eliminate the shared in-memory Repository in the worker/SQL path: construct a fresh per-activity (or per-run) Repository inside run_node — hydrate, run, sync, discard — instead of mutating one process-global instance across the 8-thread pool. The gateway/registry are cheap to rebuild per-run.**
  - 为何：Removes the single load-bearing concurrency defect (cross-run data bleed + unbounded memory leak) raised by 8/12 reviewers; without it the system cannot safely run >1 concurrent run per worker.
  - 工作量：M (touches temporal_adapter wiring + Repository scoping; ~1-2 days, plus a concurrent-runs interleave test)
- **Fix the multi-process correctness blockers as a batch: (1) require a shared S3/MinIO ephemeral tier and fail-fast at worker startup if ephemeral resolves local while runtime=temporal; (2) add .with_for_update(skip_locked=True) to the outbox dispatcher pending-select (or run it as a singleton sidecar) and move fanout to a shared bus; (3) enforce concurrency_key as a real per-provider semaphore/lease around plugin.invoke.**
  - 为何：Each independently and silently breaks the moment a second worker or API replica is added — the stated operational target. They are correctness blockers, not tuning knobs.
  - 工作量：M (ephemeral guard S, dispatcher lock S, shared-bus + provider limiter M; ~3-4 days total)
- **Run tests/integration (SQL) against a containerized Postgres in the DEFAULT CI job and add at least one backend-parity test driving a full run through both in-memory and SQL paths asserting equal RunReport/RunDetail; also add a mapper round-trip drift test.**
  - 为何：The production configuration is currently skipped by default, which is exactly why the critical wiring/concurrency bugs above survived a green 237-test suite. Cheap to stand up (docker-compose already exists) and gives a safety net for all the above refactors.
  - 工作量：S (CI wiring + ~2 parity tests; ~1 day)

### 🟡 尽快
- **Make hydrate incremental and fix the artifacts hot query: add Index(idx_artifacts_run) on ArtifactRow.run_id (and (run_id,kind)); stop the unconditional full-table SELECT of provider_profiles/voices on every node (load once per worker with TTL, or only the ids the run references); make sync write only the just-run node's delta.**
  - 为何：Per-node cost currently grows with the global catalog and worker lifetime, not the run; the hottest query degrades to seq scans at 6-figure artifact counts. Directly improves per-run latency and DB load.
  - 工作量：M (index is trivial; hydrate/sync scoping is the bulk; ~2-3 days)
- **Stream large media to/from OSS by path (boto3 upload_file/download_file into the existing disk cache) instead of read_bytes()/BytesIO, reserving put_bytes/get_bytes for small JSON artifacts; pass local cache paths directly into ffmpeg.**
  - 为何：Removes the multi-GB-per-worker transient heap and OOM path under concurrent minutes-long video, and restores the already-configured multipart concurrency.
  - 工作量：S-M (the abstraction is one method away; ~1-2 days)
- **Add heartbeat_timeout to run_node and make node execution idempotent: on entry, if a succeeded NodeRun for (run_id,node_id) exists, return its summary; otherwise reuse/patch the pending/failed row and increment attempt instead of appending. Replace the fixed 30/60s ffmpeg timeouts with duration-scaled values bounded by the Temporal activity timeout.**
  - 为何：Prevents 2-hour stalls on worker death, duplicate NodeRun rows, corrupted retry metrics, re-invocation of paid providers, and spurious render timeouts on real-length video.
  - 工作量：M (~2 days)
- **Fix the contract-layer latent defects: remove the duplicate OutboxEvent.dedupe_key redefinition (restore required:str), drop the 10 ArtifactKind silent aliases (or document them), and reconcile the 4 doubly-defined artifact classes to one canonical shape per kind. Add a ruff/test guard against duplicate field names.**
  - 为何：These are correctness bugs, not style: the nullable dedupe_key weakens at-least-once delivery, and two incompatible shapes for one artifact kind guarantee serialization mismatch between wire and persisted payload.
  - 工作量：S (~half a day + a guard test)

### 🟢 之后
- **Make engine pooling env-configurable (pool_size/max_overflow/pool_recycle/pool_timeout) sized against PG max_connections x N_workers; freeze migration 0001 to explicit op.create_table DDL with a CI autogenerate-empty-diff check.**
  - 为何：Multi-worker fan-out will contend on PG connections with no central control; create_all in 0001 silently breaks autogenerate and risks schema drift between fresh and migrated DBs.
  - 工作量：S (pooling) + M (0001 DDL snapshot + CI gate)
- **Decompose the three god-files behind stable interfaces (move ffmpeg/render + planning out of digital_human.py into packages/media/rendering and the empty packages/planning; split contracts/__init__.py into re-exported domain submodules; carve the SQL repo into per-aggregate repos), and collapse the ~259-branch dual-path into one ProductionStore interface with two implementations. Implement the empty planning package (flag the current constant-score/first-candidate/hardcoded-timing logic as the stub it is).**
  - 为何：Reduces change-amplification and the risk of latent bugs hidden by file size and SQL/in-memory drift; makes planning honestly reflect the architecture diagram. Large but mechanical and unblocked by the parity test above.
  - 工作量：L (multi-day refactor; sequence after the parity/CI safety net is in place)
